ServerManager · Feature Plan

Server Version Migration

Pick a target Minecraft version, see the ratio of mods that do and don't have a compatible build, then change the server version + move all mods + auto-disable the stragglers — in one reversible action. Works to upgrade or downgrade.

Locked decisions

chosen

Rollback

Back up first; on an unhealthy boot, restore the backup and rewrite the snapshotted DB rows. Whole move is atomic.

chosen

Scope (v1)

Change mc_version only; auto-bump fabric/quilt loader. Same platform.

chosen

Pinned mods

Moved too — pin means update cadence, not a version lock.

chosen

Incompatible

Disabled via .disabled rename (reversible), never uninstalled.

The apply — atomic migration

Snapshot rollback state

Capture mc_version/loader + every affected installed_mods row in memory — the restore target the filesystem backup can't reconstruct.

Stop server

Record prior power state so it can be restored at the end.

Back up (safety net)

Full backup via the existing C2 path; keep the backup_id. If the backup fails, abort — no net, no migration.

Re-check compatibility

Recompute server-side; don't trust the client's preview payload (builds drift).

Bump version + reinstall jar

Set mc_version (+ loader), reinstall the runtime jar via the agent.

Move mods

Compatible (incl. pinned) → swap to target build. Incompatible → rename to .disabled.

Boot & watch health

Start and watch until stable, reusing the auto-update boot watcher.

✓ Healthy

Restore prior power state. Status success (partial if some swaps failed).

✕ Unhealthy

Restore the backup, rewrite the snapshot rows + mc_version. Status failed with the boot reason.

Phased roadmap

Phase 1

Compatibility preview ships alone

Read-only endpoint + the ratio view. Useful before the apply button even exists.

  • GET /servers/{id}/mods/version-check?mc_version=X — bucket every mod, parallelized
  • loader-bump helper for fabric/quilt; validate target against mc.GameVersions
  • tests: bucketing, loader bump, validation
Phase 2

Store + run model

  • migration 010_server_version_migrations.sql
  • create/update/get/list methods mirroring ModUpdateRun
Phase 3

Migration engine

  • detached run w/ in-flight guard + progress doc
  • the 7-step apply flow + healthy/unhealthy branches above
  • tests: healthy, unhealthy→restore, backup-fail abort, mixed sets
Phase 4

API surface

  • POST /servers/{id}/migrate (settings permission)
  • GET …/migrations + …/migrations/{runId} for history/polling
Phase 5

Web UI

  • target picker + upgrade/downgrade indicator
  • ratio summary + per-bucket lists + confirm dialog
  • live run progress via polling
Phase 6

Docs

  • document the backup-restore rollback contract

What this reuses (not rebuilt)

Compatibility primitive
GetVersions(projectID, loader, mcVersion) — the preview is the existing Updates handler aimed at an arbitrary target.
Boot watch + jar swap + snapshot pattern
The autoupdate.Engine already does swapFile, startAndWatch, run-progress docs, and before-mutate row snapshots.
Runtime jar reinstall
ServerHandlers.Reinstall already stops + re-fetches the jar for the configured version.
Backup / restore + .disabled rename
The C2 backup path and the .disabled convention from SetEnabled provide the rollback + disable mechanics.
